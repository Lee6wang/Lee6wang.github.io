# 别再让桌面变成垃圾场：开发工作区分区设计与脚本化构建实践

> 你的文件系统不只是存放代码的地方，它是你整个工作方式的物理投射。如果你每次找文件都要靠搜索碰运气，每次新建项目都要纠结放哪、叫什么，那问题不在记性，而在系统。

---

## 写在前面：为什么你的电脑越用越乱

几乎每个开发者都经历过这个阶段：新电脑刚到手时干干净净，用了半年以后，桌面上几十个文件，下载目录里上千条记录，项目散落在三四个不同的父目录下，甚至还混着几个放在根目录里的 `test`、`demo`、`untitled`。你知道那个文件“大概在某个地方”，但真要找起来，往往得翻几个目录，再补两次全局搜索。

这通常不是习惯问题，而是从一开始就没有结构。

很多人用电脑的默认方式其实都差不多：桌面当入口，下载目录当仓库，文稿目录当杂物间。应付短期任务，这样当然没问题；但只要工作周期一拉长，问题就会越来越明显。原因很简单，桌面和下载目录本身没有语义，它们只是系统给你的两个默认收纳箱，无法表达“这个东西属于哪类工作”“现在处于什么阶段”。

如果你平时同时在做代码开发、文档写作、知识管理、学习笔记和内容输出，那文件系统就不是“装软件的地方”，而是**组织信息和项目的基础设施**。这个基础设施如果没设计好，后面的项目管理、知识沉淀和效率优化，基本都会变得很吃力。

所以这篇文章想解决的核心问题是：怎样给开发机器搭一套**稳定、可维护、可扩展的工作区结构**，再用脚本把它固定下来。


## 工作区分区的总体思路

### 核心原则：按职责分区，不按文件类型分

很多人整理电脑时，第一反应都是按文件类型来分：文档一个文件夹，图片一个文件夹，代码一个文件夹。这个思路看起来很合理，但实际用起来通常很快就会失效。因为一个完整项目往往同时包含代码、文档、图片和配置文件，按类型拆开之后，同一件事的上下文就被打散了。

更合适的做法是**按职责分区**。也就是让每个顶层目录代表一种工作状态，或者一类工作性质，而不是一种文件格式。

推荐的顶层结构如下：

```
~/
├── Workspace/       # 正在进行中的项目
├── Knowledge/       # 长期知识库与笔记
├── Creative/        # 创作与对外输出内容
├── Archive/         # 历史归档
└── Inbox/           # 临时接收区
```

### 各区域职责说明

**Workspace** 是主战场。所有正在推进的开发项目和工作任务都放在这里。它最重要的特征就是“活跃”，里面的内容是你最近会反复打开、反复修改的。

**Knowledge** 是知识仓库。技术笔记、读书摘录、学习心得、API 参考资料、领域知识整理，这些不一定属于某个具体项目，但以后会被反复查阅和引用的内容，都适合放在这里。

**Creative** 是创作空间。博客文章、技术演讲、开源项目文档、个人作品集，所有偏“输出”的内容，都在这里起草、修改、打磨。

**Archive** 是历史存放区。已经完成的项目、暂停的资料、不再活跃但未来可能还要翻出来的东西，都可以归档到这里。它最大的价值，就是让其他目录保持干净。

**Inbox** 是临时缓冲区。别人发来的文件、刚下载的素材、暂时还不知道该放哪儿的东西，先丢进这里。之后再定期清理，该归档的归档，该删的删。

### 为什么顶层不宜超过五六个

顶层目录是你每天打开终端或文件管理器时看到的第一层导航。如果这里放着十几个目录，你每次都得先扫一遍再决定往哪走，那本质上和没设计也差不了太多。顶层控制在五个左右，通常就足够覆盖绝大多数个人工作场景，而且每个名字一看就知道该放什么。

关键不在于分得有多细，而在于**边界要清楚**。如果你总是在两个目录之间纠结“这个东西到底该放哪”，那通常说明这两个目录的职责本身就重叠了，需要重新划边界。


## 第二层结构：让分区真正可用

顶层结构搭好之后，第二层决定了你日常操作的舒适度。

### Workspace：按项目分

```
Workspace/
├── robotics-hand/          # 灵巧手项目
├── motor-driver-stm32/     # STM32 电机驱动
├── personal-site/          # 个人网站
├── freelance-client-a/     # 客户项目
└── experiment-can-bus/     # 实验性项目
```

每个子目录就是一个独立项目，项目名就是目录名。没必要再按语言、框架或者工作性质做一层额外分类，因为“项目”本身就是最自然的组织单元。已经不再活跃的项目，就应该移到 Archive，而不是继续堆在 Workspace 里。

### Knowledge：按主题领域分

```
Knowledge/
├── embedded/               # 嵌入式开发
├── robotics/               # 机器人技术
├── programming/            # 编程通识
├── hardware/               # 硬件设计
├── tools/                  # 工具与环境
├── reading-notes/          # 读书笔记
├── papers/                 # 论文阅读
├── courses/                # 课程笔记
├── daily/                  # 日报
├── weekly/                 # 周报
└── monthly/                # 月度总结
```

这里放的是**脱离具体项目的通用知识**。比如你在做灵巧手项目时，学到了一些 I2C 通信经验，那么项目相关的实现细节可以留在 Workspace 里的项目目录里，但 I2C 协议本身的理解和笔记，更适合放进 `Knowledge/embedded/`，因为它未来很可能还会在别的项目中复用。

项目资料和长期知识最好分开存放。项目总会结束，但知识积累通常不会。如果所有笔记都散落在各个项目目录里，项目一归档，这些内容也会一起从日常视野里消失。

### Creative：按阶段分

```
Creative/
├── drafts/                 # 草稿
├── published/              # 已发布成品
├── assets/                 # 素材库（图片、模板等）
└── talks/                  # 演讲和分享材料
```

创作内容天然就有一个生命周期：素材收集、起草、修改、发布。把草稿和成品混在一起，是很常见也很容易出问题的做法。草稿属于中间态，随时会改；成品则是已经定稿的输出，应该稳定存放。如果博客草稿和已发布文章一直放在同一个目录里，时间一长，很容易就分不清哪些已经发了，哪些还在改。

### Archive：按年份分

```
Archive/
├── 2024/
├── 2025/
└── 2026/
```

归档不需要一套很复杂的分类体系。按年份分，通常就够用了。因为大多数时候，你对旧东西的记忆往往就是“那个好像是去年做的”，年份本身就是最自然的时间锚点。如果某一年的内容特别多，再在下面加子目录也不迟，不需要一开始就设计得太细。

### Inbox：保持简单

```
Inbox/
├── downloads/              # 下载缓冲
└── incoming/               # 他人发送的文件
```

Inbox 的唯一规则就是**定期清空**。它是缓冲区，不是仓库。如果一个文件在 Inbox 里放了两周以上还没处理，那通常只有两种结局：要么归档，要么删掉。


## 文件命名规则：比分类更重要的事

目录结构解决的是“放哪”的问题，命名规则解决的是“怎么找”的问题。而在真实工作里，“找”的频率通常远高于“放”。

### 为什么命名规范极其重要

macOS 的 Spotlight、Linux 下各种搜索工具、VS Code 的文件搜索，这些工具是否好用，很大程度上都取决于你的文件名质量。如果文件叫“新建文档.docx”或者“final_v2_最终版(1).pdf”，那搜索能力再强，也很难真正帮到你。

### 推荐的命名模式

对于时间相关的内容（笔记、日报、博客、归档文件）：

```
YYYY-MM-DD-主题描述.ext
```

示例：
```
2026-03-12-i2c-multiplexer-调试笔记.md
2026-03-10-seed-round-pitch-deck-v2.pptx
2026-02-灵巧手-专利申请材料.zip
```

对于项目目录，直接用简洁的英文短横线命名：

```
robotics-hand-firmware
stm32-can-driver
personal-blog-site
```

### 各类内容的命名侧重

**文档和笔记**：日期在前，主题在后。这样按文件名排序自然就是时间线顺序。

**项目目录**：用描述性的短横线命名，不加日期。项目的时间信息由 git 和文件系统自己管理。

**图片和素材**：带来源和用途描述，比如 `灵巧手-pcb-layout-v3.png` 而不是 `screenshot_2026.png`。

**归档文件**：日期 + 项目名 + 简要描述，比如 `2025-12-motor-driver-project-final.tar.gz`。

### 高风险命名方式

以下命名方式应当完全避免：

- `新建文档`、`未命名`、`test`、`demo`、`tmp` 这类无语义命名
- `最终版`、`最终版2`、`真的最终版` 这种版本失控
- 空格加中文的长文件名，容易在命令行中引发引号和转义问题
- 纯数字编号如 `001.md`、`002.md`，脱离目录上下文后完全无法辨识


## 为什么需要脚本化构建

到这里，你已经有了一套清晰的目录结构和命名规范。接下来的问题是：怎么让它稳定运转？

手动建目录最大的问题是**不一致**。今天你记得在项目里建 `docs/` 和 `references/`，下个月赶进度时可能就直接忘了。除此之外，手动操作还有一个隐性成本：每次都要重新想一遍“这次该怎么搭目录”。这种几分钟几分钟的零碎消耗，积累起来并不少。

脚本化构建的价值其实很朴素：**统一、快速、可复用**。一个 shell 函数能在零点几秒内帮你生成标准结构，比手动操作更快，也更稳定，而且每次出来的结果都一样。

shell 和 zsh 函数正好很适合干这件事。不需要额外装软件，也不用再学一门新语言，几行命令就能解决。这不是炫技，恰恰相反，它是在用最低成本处理最常见的重复劳动。


## 基础目录自动创建脚本

以下脚本用来一键初始化整个工作区结构。你只需要执行一次，就能把前面设计的五大分区及其子目录全部建好。

```bash
#!/bin/bash
# setup-workspace.sh
# 一键初始化工作区分区结构
# 用法：bash setup-workspace.sh [根目录]
# 默认根目录为 ~/

ROOT="${1:-$HOME}"

echo "🔧 正在初始化工作区结构：$ROOT"

# ===== Workspace：活跃项目区 =====
mkdir -p "$ROOT/Workspace"

# ===== Knowledge：长期知识库 =====
mkdir -p "$ROOT/Knowledge"/{embedded,robotics,programming,hardware,tools}
mkdir -p "$ROOT/Knowledge"/{reading-notes,papers,courses,daily,weekly,monthly}

# ===== Creative：创作输出区 =====
mkdir -p "$ROOT/Creative"/{drafts,published,assets,talks}

# ===== Archive：历史归档区 =====
YEAR=$(date +%Y)
mkdir -p "$ROOT/Archive/$YEAR"

# ===== Inbox：临时接收区 =====
mkdir -p "$ROOT/Inbox"/{downloads,incoming}

# 为每个顶层目录生成 README
for dir in Workspace Knowledge Creative Archive Inbox; do
  README="$ROOT/$dir/README.md"
  if [ ! -f "$README" ]; then
    echo "# $dir" > "$README"
    echo "" >> "$README"
    echo "此目录用途请参考工作区分区设计文档。" >> "$README"
  fi
done

echo "✅ 工作区初始化完成"
echo ""
echo "📂 目录结构预览："
find "$ROOT"/{Workspace,Knowledge,Creative,Archive,Inbox} -maxdepth 2 -type d 2>/dev/null | \
  sed "s|$ROOT/||" | sort | while read -r line; do
    depth=$(echo "$line" | tr -cd '/' | wc -c)
    indent=$(printf '%*s' $((depth * 2)) '')
    basename=$(basename "$line")
    echo "${indent}├── $basename/"
  done
```

这里有几个关键点值得单独说一下：

- **`mkdir -p`**：递归创建目录。如果父目录不存在会自动创建，如果目录已存在不会报错。保证幂等性——跑多少次结果都一样。
- **`README.md`**：在每个顶层目录下放一个 README，给未来的自己留下关于目录用途的说明。即使只有一行，也比空目录好得多。
- **`.gitkeep`**：如果你需要把空目录纳入 git 版本管理，可以在空目录下 `touch .gitkeep`。git 本身不追踪空目录，`.gitkeep` 是约定俗成的占位文件。


## 自定义命令体系：七大分类完整实现

接下来就是这篇文章更核心的部分了。前面的目录结构更像骨架，下面这套命令体系才是真正让它动起来的那部分。

我把自定义命令按职责拆成七类。每一类都在解决一个具体问题，加起来就是一套完整的终端工作流。

### 命令文件组织方式

在开始之前，先把命令文件本身的组织方式定好。不要把所有内容都塞进 `~/.zshrc` 里，最好按模块拆开，各自负责各自的事情：

```
~/.zsh/custom/
├── 01-navigation.zsh      # 目录导航类
├── 02-workspace-view.zsh  # 工作区查看类
├── 03-project.zsh         # 项目创建类
├── 04-knowledge.zsh       # 知识与写作模板类
├── 05-finder.zsh          # 快速打开目录类
├── 06-maintenance.zsh     # 配置与维护类
└── 07-utils.zsh           # 自动化辅助类
```

在 `~/.zshrc` 中加入自动加载逻辑：

```bash
# ~/.zshrc 末尾追加
# 自动加载所有自定义命令模块
if [ -d ~/.zsh/custom ]; then
  for config_file in ~/.zsh/custom/*.zsh(N); do
    source "$config_file"
  done
fi
```

文件名前面加数字编号，主要是为了控制加载顺序。大多数时候顺序确实没那么重要，但只要有一个模块依赖另一个模块里的变量或函数定义，顺序明确就能少掉很多莫名其妙的问题。

---

### 第一类：目录导航命令

**目标**：快速到达常用工作区，减少反复敲 `cd ~/xxx/yyy/zzz` 的重复输入。

```bash
# ~/.zsh/custom/01-navigation.zsh
# ============================================
#  目录导航命令
# ============================================

# ----- 工作区快速跳转 -----
alias ws="cd ~/Workspace && echo '📂 Workspace' && ls"
alias kn="cd ~/Knowledge && echo '📂 Knowledge' && ls"
alias cr="cd ~/Creative && echo '📂 Creative' && ls"
alias ar="cd ~/Archive && echo '📂 Archive' && ls"
alias ib="cd ~/Inbox && echo '📂 Inbox' && ls"

# ----- 通用导航 -----
alias home="cd ~ && echo '🏠 Home' && ls"
alias ..="cd .."
alias ...="cd ../.."
alias ....="cd ../../.."

# ----- 在 Finder 中打开当前目录（macOS）-----
# Linux 用户请将 open 替换为 xdg-open
alias finder="open ."
```

这些是你每天用得最多的命令。跳转之后自动 `ls` 是一个小细节——落地之后立刻看到目录内容，省去一次额外操作。`..` 和 `...` 看起来微不足道，但在深层目录里回退时体感非常好。

---

### 第二类：工作区查看命令

**目标**：快速了解当前目录体系的状态，方便检查目录是否建对、项目是否放对地方。

```bash
# ~/.zsh/custom/02-workspace-view.zsh
# ============================================
#  工作区查看命令
# ============================================

# ----- 增强版文件列表 -----
alias ll="ls -lhF"        # 详细列表，人类可读大小
alias la="ls -lhFA"       # 同上，包含隐藏文件
alias lt="ls -lhFt"       # 按修改时间排序（最新在前）

# ----- 树状结构查看 -----
# showtree：查看指定目录的树状结构
# 用法：showtree [目录] [层数]
# 示例：showtree ~/Knowledge 2
showtree() {
  local target="${1:-.}"
  local depth="${2:-2}"

  # 优先使用 tree 命令（如果已安装）
  if command -v tree &> /dev/null; then
    tree -L "$depth" -d --charset utf-8 "$target"
  else
    # 回退方案：用 find 模拟
    echo "📂 $target (深度: $depth)"
    find "$target" -maxdepth "$depth" -type d ! -path '*/\.*' 2>/dev/null | \
      sed "s|$target||" | sort | while read -r line; do
        [ -z "$line" ] && continue
        depth_count=$(echo "$line" | tr -cd '/' | wc -c)
        indent=$(printf '%*s' $((depth_count * 2)) '')
        echo "${indent}├── $(basename "$line")/"
      done
  fi
}

# ----- 顶层全景 -----
# overview：一次性查看所有工作区的顶层结构
overview() {
  echo "========================================="
  echo "  📊 工作区全景"
  echo "========================================="
  echo ""
  for dir in Workspace Knowledge Creative Archive Inbox; do
    local full_path="$HOME/$dir"
    if [ -d "$full_path" ]; then
      local count=$(ls -1 "$full_path" 2>/dev/null | wc -l | tr -d ' ')
      echo "📂 $dir/ ($count 项)"
      ls -1 "$full_path" 2>/dev/null | head -8 | while read -r item; do
        echo "   ├── $item"
      done
      [ "$count" -gt 8 ] && echo "   └── ... (还有 $((count - 8)) 项)"
      echo ""
    fi
  done
}
```

`showtree` 是个很实用的函数。它优先使用 `tree` 命令（如果你装了的话，macOS 上 `brew install tree` 即可），没装就用 `find` 模拟一个。`overview` 可以让你在一个屏幕里看到所有工作区的概况——适合每天早上打开终端时瞄一眼。

### 第三类：项目创建命令

**目标**：一键创建统一的新项目结构，保证每个项目从一开始就骨架完整。

```bash
# ~/.zsh/custom/03-project.zsh
# ============================================
#  项目创建命令
# ============================================

# mkproj：创建标准项目结构
# 用法：mkproj <项目名>
# 示例：mkproj robotics-hand-v2
mkproj() {
  if [ -z "$1" ]; then
    echo "用法：mkproj <项目名>"
    echo "示例：mkproj robotics-hand-v2"
    return 1
  fi

  local project_name="$1"
  local project_root="$HOME/Workspace/$project_name"

  # 防止重复创建
  if [ -d "$project_root" ]; then
    echo "⚠️  项目目录已存在：$project_root"
    echo "   如需进入该项目，请执行：cd $project_root"
    return 1
  fi

  # 创建项目子目录
  mkdir -p "$project_root"/{docs,src,assets,references,output,temp}

  # 生成项目 README
  cat > "$project_root/README.md" << EOF
# $project_name

**创建时间**：$(date +%Y-%m-%d)
**状态**：进行中
```

### 为什么这些子目录值得存在

**`output/`**：构建产物和最终交付物最好单独存放。代码在 `src/`，产出在 `output/`，不要混在一起。对嵌入式项目来说，编译出来的 `.bin`、`.hex` 文件单放在 `output/` 里会非常清楚。

**`temp/`**：临时文件永远都会有，比如调试日志、实验数据、中间产物。与其让它们散落在项目根目录里，不如明确给它们一个位置，再通过 `.gitignore` 排除出版本管理。

**`references/`**：数据手册、芯片参考文档、竞品分析材料，这些东西不是你写的，但对项目很重要。跟自己的文档（`docs/`）分开放，查找时会清楚很多。

**`README.md`** 在项目初期最重要的作用，其实不是写给别人看，而是写给未来的自己。项目刚开始时，你脑子里全是上下文；但三个月后再回来，很多当时觉得“不可能忘”的东西，往往已经模糊了。README 哪怕只有三行，项目名、创建日期、一句话目标，也足够帮你快速恢复现场。

### 模板化的核心价值

模板解决的不是“排版好不好看”的问题，而是**把记录这件事的门槛降下来**。很多时候，人不是没东西写，而是一打开空白文件就卡住了，不知道从哪一句开始。模板的作用就是跳过这一步：文件已经建好，结构已经摆好，你只需要往里填内容。

**固定位置 + 固定命名 + 固定结构**，这三者叠在一起的效果非常强。你的笔记、日报、博客草稿、论文记录都会变得可预测。你知道它们在哪，知道它们叫什么，也知道打开之后大概会看到什么。这种可预测性对长期积累非常重要，因为半年以后你再回头翻，不需要重新猜测每个文件是干什么的。

### 第七类：自动化辅助命令

**目标**：减少机械重复动作，提供一些通用的小工具函数。

```bash
# ~/.zsh/custom/07-utils.zsh
# ============================================
#  自动化辅助命令
# ============================================

# ----- mkgitkeep：为空目录添加 .gitkeep -----
# 用法：mkgitkeep [目录]
# 不指定目录时，为当前目录下所有空子目录添加 .gitkeep
mkgitkeep() {
  if [ -n "$1" ]; then
    # 指定目录：只为该目录添加
    mkdir -p "$1"
    touch "$1/.gitkeep"
    echo "✅ 已添加：$1/.gitkeep"
  else
    # 未指定：扫描当前目录下所有空子目录
    local count=0
    find . -type d -empty ! -path '*/\.*' | while read -r dir; do
      touch "$dir/.gitkeep"
      echo "  + $dir/.gitkeep"
      count=$((count + 1))
    done
    echo "✅ 共处理 $count 个空目录"
  fi
}

# ----- mkreadme：在当前目录生成 README.md -----
# 用法：mkreadme [标题]
# 不指定标题时，使用当前目录名
mkreadme() {
  local title="${1:-$(basename "$(pwd)")}"
  local filepath="./README.md"

  if [ -f "$filepath" ]; then
    echo "⚠️  README.md 已存在"
    ${EDITOR:-code} "$filepath"
    return
  fi

  cat > "$filepath" << EOF
# $title

**创建日期**：$(date +%Y-%m-%d)

## 简介

## 目录说明

## 备注

EOF

  echo "📄 README.md 已创建"
  ${EDITOR:-code} "$filepath"
}

# ----- today：输出今日日期字符串 -----
# 方便在其他命令中使用，或直接复制
today() {
  echo "$(date +%Y-%m-%d)"
}

# ----- confirm：通用确认提示 -----
# 内部工具函数，供其他脚本调用
confirm() {
  local prompt="${1:-确认执行？}"
  echo -n "$prompt [y/N] "
  read -r answer
  [[ "$answer" =~ ^[Yy]$ ]]
}
```

这类命令单看每一个都不大，但日常出现频率其实很高。`mkgitkeep` 解决的是 git 不追踪空目录的老问题；`mkreadme` 让你在任何目录下一秒钟就能生成一个标准 README；`today` 看上去简单得像是多余，但真到你需要在终端里拼文件名的时候，`$(today)` 就会比手打一遍日期顺手很多。


## 工作区分区与脚本化构建如何形成闭环

到这里，可以回头看一下整套体系是怎么连起来的：

**目录结构**解决的是"放哪"的问题——每一类内容都有明确的归属区域，不用每次都做决策。

**命名规则**解决的是"怎么找"的问题——稳定的命名模式让搜索变得可靠，让按名排序变得有意义。

**自动化命令**解决的是"怎么快速创建"的问题——从想法到落地之间不需要手动搭脚手架。

**模板**解决的是"怎么开始"的问题——打开文件就能动手写，不用对着空白页发呆。

这四件事放在一起，才算是一套完整工作流。缺了目录结构，文件就会散；缺了命名规则，找东西就只能碰运气；缺了自动化，每次都要重复搭脚手架；缺了模板，很多记录根本启动不起来。

更重要的是，这个体系不是静止摆在那儿的，而是**动态运转**的。项目在 Workspace 里开始，知识沉淀到 Knowledge，创作成果在 Creative 里成形，完成后的东西归到 Archive，新来的内容先进入 Inbox 再分流。文件会在这些区域之间流动，而不是长期堆在某个角落里慢慢失控。

```
Inbox (收) → Workspace (做) → Creative (出) → Archive (存)
              ↕
          Knowledge (学)
```

这不是一张静态目录图，而更像是你日常工作的路线图。


## 现实提醒与边界

真正开始动手之前，还有几点很实际的提醒：

**不要一开始就设计得太复杂。** 五个顶层目录加上各自一两层子目录就足够了。过度设计的目录体系跟过度设计的代码一样——你会因为维护成本太高而放弃使用它。

**不要为了自动化而自动化。** 如果你一个月只建一次项目，手动 mkdir 完全可以接受。脚本化的前提是"这件事我经常做，并且每次都差不多"。如果一个操作你一年只做两次，手动做就行。

**不要追求完美分类。** 总会有些文件你不确定该放哪，这很正常。Inbox 就是为此存在的。重要的不是每个文件都完美归位，而是大部分文件都在可预期的位置。

**真正重要的是坚持使用。** 一套普通但你每天都在用的系统，远比一套精妙但三天后就弃用的系统有价值。选择你能坚持的复杂度，然后执行下去。

**命令要渐进式添加。** 不需要第一天就把七类命令全部配齐。先从最高频的开始——导航命令和 `mkproj` 可能是最先值得有的，用着用着自然知道还需要什么。


## 结语

你搭起来的，其实不只是一组文件夹，而是一套**个人工作基础设施**。

代码项目在 Workspace 里推进，技术积累在 Knowledge 里沉淀，创作内容在 Creative 里从草稿走向发布，做完的工作则进入 Archive，安静待着，等以后需要时再被找出来。你在终端里敲下的每一条命令，比如 `mkproj`、`mknote`、`mkdaily`、`mkweekly`、`mkmeeting`，本质上都是在给自己造工具，让日常工作从“先想清楚该怎么放”变成“直接开始做”。

好的工作区分区设计，最后会让你的电脑变成一个越用越顺手、越用越有序的地方。这件事很值得花一个下午认真做，因为它带来的回报，往往会持续很多年。
