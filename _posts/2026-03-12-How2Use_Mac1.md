# 新 MacBook 到手，我是这样一步步搭出长期开发环境的

## 写在前面

每次拿到一台新 MacBook，打开盖子的那一刻都挺让人兴奋的：系统是干净的，桌面是空的，也没有任何历史遗留问题。但对开发者来说，这种“清爽”很快就会变成另一件事: 你得重新把一整套工作环境搭起来。

很多人会把这件事理解成“装几个软件”。去 App Store 下一圈，再从官网拖几个 `.dmg`，差不多就觉得结束了。但如果这台机器要陪你工作一两年，甚至更久，你很快就会发现，真正麻烦的从来不是“装软件”，而是把自己的工作流重新安顿好。工具链怎么排，项目目录怎么放，终端怎么配，哪些事情值得顺手自动化，这些才是真正影响日常效率的地方。

这篇文章写的，就是我自己在一台全新 MacBook 上，从零把开发环境搭起来的过程。我的使用场景大致是这些：

- **代码开发**：C/C++、Python、偶尔写点前端脚本
- **嵌入式辅助开发**：通过 Mac 做编译、烧录、串口调试，连接 STM32 等开发板
- **SSH 远程开发**：连到远程 Linux 服务器做编译和部署
- **脚本和工具链**：日常写一些自动化脚本、数据处理、构建流程
- **文档和知识管理**：写技术博客、整理项目文档、管理个人知识库

我想搭出来的，不是一台“软件装得很全的电脑”，而是一个能长期支撑学习、开发和输出的工作平台。

---

## 先明确目标：这台机器要承担什么角色

在动手装任何东西之前，我建议先停下来想清楚一个问题：**这台 MacBook 在你的工作流里，到底要扮演什么角色？**

这个问题听上去有点虚，但它会直接决定你后面的大多数选择。比如：

- 如果你主要做 Web 全栈，那 Node.js 生态和容器化工具是核心；
- 如果你做嵌入式，那串口工具、交叉编译链和远程调试的优先级更高；
- 如果你还要做内容输出，那文档工具和目录结构也不能随意。

我给自己的定位是：**这是一台现代开发工作流里的主力机**。它不需要什么都能跑，某些厂商专有 IDE、特定的 Windows 驱动、老旧仿真工具，我并不强求一定在 Mac 上解决。但它至少要能覆盖我日常 80% 以上的开发和写作需求，而且用起来要顺手。

换句话说，我追求的不是“万能”，而是“高效，而且能长期用下去”。

## 系统级基础环境：地基要先打好

### 更新系统

拿到新机器后，第一件事就是先把 macOS 更新到最新版本。这不是强迫症发作，而是后面很多工具对系统版本都有最低要求，Homebrew 和 Xcode CLI Tools 也都依赖比较新的系统组件。

打开"系统设置 → 通用 → 软件更新"，等它跑完就好。

### 安装 Xcode Command Line Tools

macOS 默认并不附带完整的编译工具链。你在终端里敲 `gcc` 或 `git`，大概率都会先提示你安装这个。它提供了 GCC/Clang 编译器、Git、Make 等基础工具，后面很多开发工具都要靠它打底。

```bash
xcode-select --install
```

弹出对话框后点安装就行，一般几分钟就能搞定。装完之后可以顺手验证一下：

```bash
gcc --version
git --version
```

这里装的不是完整的 Xcode IDE（那个体积很大），只是命令行工具集而已。如果你不做 iOS 开发，完整 Xcode 完全可以先不装。

### 安装 Homebrew

Homebrew 基本就是 macOS 上默认的包管理器。在 Linux 上你用 `apt` 或 `yum`，在 Mac 上对应的就是 `brew`。后面要装的大多数命令行工具、开发库，甚至一部分 GUI 应用，都可以交给它来管。

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

安装完成后，按终端提示把 Homebrew 加进 PATH。Apple Silicon 的 Mac 上，Homebrew 默认安装在 `/opt/homebrew`，通常要在 shell 配置里补上：

```bash
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc
```

验证一下：

```bash
brew --version
```

为什么我会把 Xcode CLI Tools 和 Homebrew 放在最前面？因为它们本身不是具体的开发工具，而是**后面所有工具的基础设施**。这两个没处理好，后面的步骤基本都会卡。

## 通用开发工具：先把工具箱填满

基础环境就位之后，接下来就是把一批几乎所有开发者都会用到的通用工具装上。我的原则很简单：**先装高频刚需，不一次性堆太多“可能以后会用”的东西。**

### Git

虽然 Xcode CLI Tools 自带 Git，但版本通常偏旧。用 Homebrew 装一个更新的版本，会省掉后面不少小问题：

```bash
brew install git
```

装完之后，先把最基本的配置补上：

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
git config --global init.defaultBranch main
```

如果你平时用 GitHub，建议顺手把 SSH key 也配好，省得每次 `push` 都输密码：

```bash
ssh-keygen -t ed25519 -C "你的邮箱"
cat ~/.ssh/id_ed25519.pub
# 复制公钥，添加到 GitHub → Settings → SSH Keys
```

### CMake 和 Ninja

如果你写 C/C++，CMake 基本绕不过去。Ninja 则是一个轻量、很快的构建执行器，和 CMake 搭配起来，编译体验会明显好不少。

```bash
brew install cmake ninja
```

### Python

macOS 自带的 Python 版本通常比较老，也不太适合直接拿来做开发。用 Homebrew 装一个独立的 Python，会干净很多：

```bash
brew install python
```

装完之后，`python3` 和 `pip3` 就能直接用了。如果后面项目需要管理多个 Python 版本，再补 `pyenv` 也不迟，一开始没必要把事情搞复杂。

有一点我很建议尽早养成习惯：用虚拟环境。一个项目一个 `venv`，能避开大多数全局包冲突问题：

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### Node.js

即使你不是前端开发者，Node.js 现在也很难完全绕开。很多 CLI 工具、文档生成器、Lint 工具，甚至一些 AI 工具链，背后都依赖它。我的建议是直接用 `nvm` 管版本：

```bash
brew install nvm
```

然后在 `~/.zshrc` 里加上 nvm 的初始化配置（安装时终端会给提示），之后：

```bash
nvm install --lts
nvm use --lts
node --version
```

### VS Code

Visual Studio Code 现在基本就是最通用的跨平台代码编辑器了。生态够成熟，插件也够全，从 C/C++ 到 Python，再到远程开发，基本都能覆盖。

```bash
brew install --cask visual-studio-code
```

装完以后，我一般会先补这几个核心插件：

- **C/C++**（微软官方）：语法高亮、IntelliSense、调试支持
- **Python**（微软官方）：Python 开发全套支持
- **Remote - SSH**：连接远程服务器，在本地 VS Code 里编辑远程代码
- **GitLens**：增强 Git 历史查看和 blame 功能
- **Markdown All in One**：写文档和博客时很方便

VS Code 还可以直接从命令行启动。在终端里敲一个 `code .`，当前目录就能直接打开，日常用起来很顺。如果这个命令还不能用，可以在 VS Code 里按 `Cmd+Shift+P`，搜索"Shell Command: Install 'code' command in PATH"启用。

### iTerm2

macOS 自带的 Terminal 当然能用，但功能还是比较基础。iTerm2 这些年基本成了很多 Mac 用户的默认选择，分屏、标签页、快捷键自定义、配色这些都更成熟，长期用下来体验确实好很多。

```bash
brew install --cask iterm2
```

再配一个顺手的 shell 环境，终端体验会舒服不少。我通常会用 Oh My Zsh 来管理 zsh 配置：

```bash
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"
```

### 其他常用命令行工具

下面这些工具我建议一口气装好，后面用到的频率非常高：

```bash
brew install tmux wget curl htop tree jq ripgrep fd bat
```

简单说一下我为什么会保留这些：

- **tmux**：终端复用器，SSH 到远程服务器时尤其有用，断线也不怕丢会话
- **wget / curl**：下载文件和调试 HTTP 请求
- **htop**：比系统自带的 `top` 好用得多的进程监控工具
- **tree**：以树状结构查看目录，了解项目结构很方便
- **jq**：命令行 JSON 处理工具，调试 API 返回数据时离不开
- **ripgrep（rg）**：比 grep 快很多的全文搜索工具
- **fd**：比 find 好用的文件查找工具
- **bat**：带语法高亮的 `cat` 替代品

这些工具单看都不大，但一旦组合起来，命令行环境会顺手很多，很多零碎操作都能明显提速。

## 与个人开发方向相关的补充工具

上面这些属于通用配置。接下来这一部分，就比较看个人方向了。我的工作偏嵌入式和硬件相关开发，所以补充工具也主要围绕这个场景来。

### 串口调试

做嵌入式开发，串口几乎是最基础的调试手段。Mac 上常用的串口工具有 `minicom` 和 `screen`：

```bash
brew install minicom
```

插上 USB 转串口线之后，可以先用下面的命令看看设备有没有识别出来：

```bash
ls /dev/tty.usb*
```

然后连接串口：

```bash
minicom -D /dev/tty.usbserial-xxxx -b 115200
```

如果你更喜欢图形界面，CoolTerm 也不错，直接从官网下载就行。

### OpenOCD

OpenOCD 是开源的片上调试工具，支持 JTAG/SWD 协议，配合 ST-Link、J-Link 等调试器可以给 STM32 等芯片烧录和调试：

```bash
brew install open-ocd
```

不过这里要提前打个预防针：OpenOCD 在 Mac 上偶尔会遇到驱动兼容性问题，尤其是某些调试器还需要额外装 USB 驱动。真碰到问题时，通常还是得回到具体调试器的官方文档里找答案。

### ARM 交叉编译工具链

如果你打算在 Mac 上编译 ARM Cortex-M 系列固件，那 ARM GCC 工具链基本也是必装项：

```bash
brew install --cask gcc-arm-embedded
```

装完后验证：

```bash
arm-none-eabi-gcc --version
```

配合 CMake 和 Ninja，大部分嵌入式项目的编译工作就都能在 Mac 上完成了。

### Mac 在嵌入式开发中的优势与边界

说实话，Mac 在嵌入式开发这件事上，并不是传统意义上的主力平台。很多厂商 IDE，比如 Keil MDK、IAR，都只支持 Windows；一些专有烧录工具和驱动，也基本是 Windows 独占。

但它也不是不能做，反过来看，Mac 的优势其实也很明确：

- **现代开发流程兼容性好**：GCC + CMake + OpenOCD 这套开源工具链在 Mac 上运行流畅，适合走 Makefile/CMake 构建路线的项目
- **终端体验远好于 Windows**：zsh + iTerm2 + tmux 的组合，日常操作效率很高
- **SSH 远程开发能力强**：连接 Linux 编译服务器非常方便
- **和其他开发工作无缝切换**：写完固件代码，切到 Python 写个测试脚本，再切到 Markdown 写文档，不需要换机器

我自己的策略一直很明确：**Mac 负责日常编码、编译、调试和文档；一旦遇到必须依赖 Windows 专有工具的场景，就开虚拟机，或者直接换另一台 Windows 机器处理。** 没必要硬把所有事情都塞进一台机器里。

## 工作区与目录结构：不要忽视文件体系设计

很多人搭环境时，眼里只有“装了什么软件”，但还有一件同样重要的事常常被忽略：**文件到底放哪儿，怎么组织。**

刚开始用的时候，这件事看起来好像无所谓；但过上半年你就会发现，项目散在桌面、下载文件夹和各种临时目录里，想找个东西得翻很久。目录结构如果一开始就设计清楚，后面能省掉很多心智负担。

我现在常用的目录结构大概是这样：

```
~/
├── Workspace/          # 所有代码项目
│   ├── embedded/       # 嵌入式相关项目
│   ├── python/         # Python 项目
│   ├── scripts/        # 工具脚本
│   ├── web/            # 前端 / Web 项目
│   └── playground/     # 临时实验和学习
│
├── Knowledge/          # 知识库和学习笔记
│   ├── notes/          # 技术笔记
│   ├── references/     # 参考资料和手册
│   └── reading/        # 读书笔记
│
├── Creative/           # 内容创作输出
│   ├── blog/           # 博客文章
│   ├── docs/           # 对外文档
│   └── slides/         # 演示文稿
│
├── Archive/            # 归档（完成或暂停的项目）
│
└── Inbox/              # 临时收纳（待整理的文件）
```

这里面有几个我自己比较在意的设计思路：

- **Workspace 按技术方向分子目录**，而不是按项目名随意堆放。这样你能快速定位某类项目在哪里。
- **Knowledge 和 Creative 分开**。前者是输入（学习、积累），后者是输出（写作、发布）。混在一起会让你分不清哪些内容是给自己看的，哪些是要对外的。
- **Inbox 是一个缓冲区**。临时下载的文件、别人发来的资料、还没想好放哪儿的东西，先扔 Inbox，定期整理归位。
- **Archive 是"冷存储"**。完成的项目、不再活跃的代码，移到这里，不删但也不占主目录的视线。

可以用一行命令把这个结构建好：

```bash
mkdir -p ~/Workspace/{embedded,python,scripts,web,playground} \
         ~/Knowledge/{notes,references,reading} \
         ~/Creative/{blog,docs,slides} \
         ~/Archive \
         ~/Inbox
```

这不是什么高深技巧，但它确实是**让你的机器从“一堆文件”变成“有组织的工作平台”的关键一步**。

## 环境搭建完成后，这台机器能做什么

到这里，环境基本就搭得差不多了。回头看一眼，这台 MacBook 现在大概能承接这些工作：

**写 C/C++ 代码**：有 GCC/Clang 编译器、CMake + Ninja 构建系统、VS Code 的 C/C++ 插件提供智能补全和调试支持。无论是本地项目还是嵌入式交叉编译，都能覆盖。

**写 Python**：有独立的 Python 环境、venv 虚拟环境管理、pip 包管理。从数据处理脚本到自动化工具到简单的 Web 服务，都没问题。

**做工具脚本**：有完善的 shell 环境、丰富的命令行工具（jq、ripgrep、fd 等），再加上 Python，足以应对日常的自动化、数据处理、批量操作需求。

**管理项目**：有 Git 做版本管理、清晰的目录结构做项目分类、VS Code 做代码浏览和编辑。多个项目并行推进时不会混乱。

**连接开发板**：有串口工具 minicom、调试工具 OpenOCD、ARM 交叉编译工具链。插上开发板就能编译、烧录、调试。

**远程 SSH 开发**：有 iTerm2 + tmux 做终端管理、VS Code Remote SSH 做远程编辑。连到远程 Linux 服务器的体验几乎和本地一样。

**写文档和博客**：有结构化的知识库目录、Markdown 编辑支持、笔记快速创建模板。从技术笔记到正式博客文章，都有地方安放。

它看起来像是一串工具清单，但更准确地说，这是一套能彼此配合起来的工作流。

## 现实问题与边界：Mac 不是万能答案

客观来说，Mac 在开发领域确实有很多优势，但边界也很明确。如果不正视这些边界，最后大概率是在给自己加额外麻烦。

**厂商专有工具依赖 Windows**。嵌入式领域最典型：Keil MDK、IAR Embedded Workbench 都只有 Windows 版本。一些芯片厂商的配置工具（比如某些 MCU 的时钟配置器、引脚分配工具）也是 Windows 独占。如果你的项目强依赖这些工具，那就绕不开 Windows。

**某些硬件驱动支持不完善**。虽然大部分 USB 转串口芯片（CH340、CP2102、FTDI 系列）在 Mac 上都有驱动，但偶尔还是会遇到某个调试器或编程器在 Mac 上识别不了的情况。

**虚拟机和双系统有额外成本**。Apple Silicon Mac 上跑 Windows 虚拟机（通过 Parallels 或 UTM）是可行的，但性能和兼容性不如原生 Windows。ARM 版 Windows 在某些 x86 应用的兼容性上还有缺口。

**我的应对策略是三层结构**：

1. **Mac 做主力**：日常编码、编译、调试、文档、博客，全在 Mac 上完成
2. **虚拟机做补充**：偶尔需要用的 Windows 工具，在虚拟机里解决
3. **独立 Windows 机器兜底**：真正重度依赖 Windows 的场景（比如某些量产烧录工具），用专门的 Windows 机器处理

不要执着于让一台机器包办所有事情。认清每种工具擅长做什么，再把任务分配清楚，通常才是效率最高的做法。

## 结语

回头看这篇文章，从系统更新、Homebrew 安装，一路写到目录结构和终端环境，内容确实不少。但如果让我用一句话概括这件事的核心，其实就是：

**你搭建的不是一台装满软件的电脑，而是一个能够长期承接学习、开发、写作与输出的工作系统。**

软件会更新，工具会替换，但你慢慢建立起来的工作流、目录结构和使用习惯，是会一直跟着你走的。它们才是真正意义上的“环境”。

每个人的开发方向和使用习惯都不一样，我这里分享的，也只是我自己用下来比较顺的一套方案。你完全可以按自己的需求去调整，删掉不需要的部分，再补上你自己的工具和配置。重要的是，**从一开始就有意识地设计这套环境，而不是用到哪儿装到哪儿，最后把机器用成一团乱麻。**

希望这篇文章对正在配置新 MacBook 的你有所帮助。如果你也有自己的环境搭建心得，欢迎交流。
