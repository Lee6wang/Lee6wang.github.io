# Understanding Time in an RTOS: The 10 ms Task You Thought Was Precise

> If you have already started using FreeRTOS, you have probably written code like this:
>
> ```c
> for (;;)
> {
>     do_work();
>     osDelay(10);
> }
> ```
>
> It looks very reasonable, and it does indeed execute "about every 10 ms."
>
> But as long as the system runs long enough and the load becomes a little more complex, you will find that:
>
> - The task rhythm starts to drift
> - Multiple periodic tasks gradually lose alignment
> - Some functions that were "supposed to be stable" occasionally become abnormal
>
> The problem is often not in the algorithm, or even in the business logic.
> **It is that you used the RTOS time model incorrectly from the beginning.**

In this article, we will do only one thing:

> **Explain thoroughly what "time" inside an RTOS really is,
> and how, in engineering practice, to write periodic tasks that do not drift away.**

---

## 1. An Intuition That Must Be Corrected First

In bare-metal development, our understanding of "time" is familiar:

> "Delay 10 ms -> the next execution happens 10 ms later."

But in an RTOS, this intuition is **not valid**.

In FreeRTOS, most time-related APIs do not describe:

> "Execute precisely at a certain time point"

Instead, they describe:

> **"After waiting at least this long, the task has a chance to continue running."**

This subtle-looking difference is the root of all the later problems.

---

## 2. The Source of All "Time" in an RTOS: Tick

FreeRTOS does not know what a "millisecond" is.
The only unit of time it understands is: **Tick**.

- Tick is generated by an interrupt with a fixed frequency, often configured as 1 ms
- On every Tick interrupt:
  - The system Tick count increments
  - The kernel checks whether delayed or blocked tasks have expired
  - A scheduling operation is triggered when necessary

From the RTOS point of view:

> **Time is not a continuous flow. It jumps forward one grid at a time.**

This means a very important but often ignored fact:

> **Time inside an RTOS is essentially discrete and approximate, not continuously precise.**

---

## 3. The Most Common and Most Dangerous Pattern

When writing periodic tasks, many people naturally use the following structure:

```c
for (;;)
{
    do_work();
    osDelay(10);
}
```

The problem is not "right now."
The problem is **what happens after the system has run long enough**.

---

### 3.1 What Is Its Real Execution Period?

The actual period of one loop is approximately:

```c
do_work() execution time + task rescheduling delay + osDelay(10)
```

As long as the execution time of `do_work()` is not **strictly 0**, the real period of this task must be **greater than 10 ms**.

Even more critically:

- Every round adds a little error
- The error accumulates in only one direction
- The task rhythm slowly but irreversibly drifts backward

---

### 3.2 A Very Realistic Engineering Scenario

Suppose:

- You designed a **10 ms control task**
- The average execution time of `do_work()` is about 2 ms

Then:

- Designed period: 10 ms
- Actual period: about 12 ms

The result is:

- After 1 minute: about **120 ms** slower
- After 10 minutes: **1.2 seconds** slower

In control, communication, and synchronization systems, this is already a **disaster-level consequence**.

---

## 4. What Does a "Real Periodic Task" Look Like in an RTOS?

In FreeRTOS, a **periodic task is not implemented by "delaying"**.
It is implemented by **aligning to a time reference**.

This is exactly why `vTaskDelayUntil()` exists.

---

### 4.1 The Correct Engineering Model for a Periodic Task

```c
void PeriodicTask(void *arg)
{
    TickType_t last_wake_time = xTaskGetTickCount();

    for (;;)
    {
        do_work();
        vTaskDelayUntil(&last_wake_time, pdMS_TO_TICKS(10));
    }
}
```

The core idea behind this code is only one sentence:

> **The task is not "finish work and then wait 10 ms."
> It is "always align to a 10 ms time grid."**

---

### 4.2 Why Does It Not Drift?

The key is the meaning of `last_wake_time`.

It does not record:

> "The time when the task last finished"

It records:

> **"The previously planned wake-up time."**

FreeRTOS behaves like this:

- No matter how long `do_work()` actually took
- As long as it did not exceed one period
- The next wake-up time returns to the original rhythm line

This is a **real periodic task** in the engineering sense.

---

## 5. What Happens If Task Execution Time Exceeds the Period?

This is a problem that **must be faced directly but is often avoided**.

Suppose:

- Period = 10 ms
- Actual execution time = 15 ms

Then the result is:

- The RTOS will not "save you"
- The task will run again immediately at the next scheduling point
- The system enters a continuous state of "chasing time"

The engineering conclusion is very clear:

> **The RTOS can only guarantee correct scheduling logic.
> It cannot solve insufficient computing power for you.**

Once a periodic task times out for a long period, you must actively adjust the design:

- Lower the task frequency
- Split the task structure
- Or optimize at the algorithm level

---

## 6. `timeout` Is Not `delay`: Another Frequent Misunderstanding

Look at this line of code:

```c
xQueueReceive(queue, &msg, pdMS_TO_TICKS(100));
```

Many people instinctively interpret it as:

> "If there is no data within 100 ms, execute again after 100 ms."

But its real meaning is:

> **"I will wait at most 100 ms.
> If data arrives within 1 ms, I wake immediately."**

In other words:

- `timeout` describes the **maximum waiting time**
- It is not a fixed delay or period

Using `timeout` as `delay` is a **hidden but extremely common mistake** in RTOS systems.

---

## 7. What Is `osDelay(0)`? Why Is It Almost Useless?

```c
osDelay(0);
```

This line does not truly block the task.

Its behavior is only:

- If there is a Ready task with the **same priority**, yield the CPU once
- If there is not, continue executing

The engineering conclusion is direct:

> **`delay(0)` cannot be used as a task blocking method.**

If you see large amounts of `osDelay(0)` in a system, it is usually an **architecture-level warning signal**.

---

## 8. Four Iron Rules for Engineering-Level Use of Time

If you remember only one part of this article, I hope it is these four rules:

1. **Strict periodic tasks must use `vTaskDelayUntil`**
2. **`vTaskDelay / osDelay` is only for tasks without strict rhythm**
3. **`timeout` means "how long to wait at most," not "execute after how long"**
4. **`delay(0)` is not blocking; it is only one yield**

Follow these four rules, and you can avoid most **hidden time-related bugs**.

---

## 9. Put Time Problems Back Into the Overall RTOS View

Returning to the sentence mentioned in the first article:

> **FreeRTOS is a "system of order,"
> not a "time magic system."**

The RTOS can help you with only three things:

- Provide a unified, discrete time scale
- Make predictable scheduling decisions at time points
- Ensure deterministic system behavior

But the prerequisite is:

> **You must understand and use time in the RTOS way.**
