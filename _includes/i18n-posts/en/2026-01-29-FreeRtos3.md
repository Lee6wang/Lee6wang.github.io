# The Memory World of an RTOS: Heap, Stack, and System Stability

**- Why the system "dies after running for a while" even though you did nothing obviously wrong**

Anyone who has built a slightly complex project with FreeRTOS has probably experienced this:

- The system starts normally
- Logs print correctly and tasks are running
- The functions also appear to work
- But after running for some time, it suddenly HardFaults
- Or some task silently stops executing forever

What is even more frustrating is that there is no obvious logic error, no visible array overflow, and even single-step debugging makes everything "look correct."

If you have experienced this, the problem may not be in the business logic at all. It may be in the **memory world** of the RTOS.

This article does not discuss algorithms or communication. It focuses on one thing: which memory areas an RTOS system is actually using, and how system stability is unintentionally damaged.

------

## 1. First Build the Full Picture

In systems such as FreeRTOS + STM32, at least **four completely different kinds of memory** exist:

| Memory area | Purpose |
| ----------- | ------- |
| Flash | Program code and constants |
| RTOS Heap | Kernel objects such as tasks, queues, and semaphores |
| Task Stack | The private stack of each task |
| MSP (interrupt stack) | Shared by all interrupts |

If you simply call all of them "RAM" in a project, sooner or later you will pay for it, because these four memory areas have different users, different lifetimes, and different symptoms when they fail.

------

## 2. RTOS Heap: Not "Your Memory," But "Kernel Memory"

This is the most easily misunderstood area.

**The RTOS Heap is responsible for only one thing: allocating memory for kernel objects.** These include Task objects (TCB + stack), Queue, Semaphore, Mutex, Event Group, and Software Timer.

It is **not used for** local variables inside tasks, ordinary global variables, or `malloc` in business code unless you use `heap_3`.

One key point is: **RTOS Heap exhaustion often does not immediately crash the system.** More commonly, creating a task or queue fails and returns `NULL`, but the code does not check it. The system continues running and eventually crashes in a place that seems completely unrelated. This type of problem is among the hardest to locate and easiest to misjudge in engineering.

------

## 3. Task Stack: The Real "First Accident Scene"

If RTOS Heap is a chronic risk, then **task stack is the most common accident scene**.

Each task owns an independent stack used to store local variables, function call context, interrupt return context, and temporary buffers from `printf` / `snprintf`. It is completely separate from the RTOS Heap.

Consider an extremely realistic beginner stack-explosion scenario:

```c
void Task(void *arg)
{
    uint8_t buf[1024];
    for (;;)
    {
        process(buf);
        osDelay(100);
    }
}
```

If this task stack is configured as 256 words, which is roughly 1 KB, then this one array already consumes the stack. Add function calls and `printf`, and stack overflow is almost inevitable. The result is often not immediate death, but running for a while, random HardFaults, and debugging that gives no clear clue.

------

## 4. Why Is Stack Overflow So Hard to Locate?

This is one of the most painful points for RTOS beginners, and the reason is simple:

Stacks grow downward. When overflow happens, it overwrites other tasks' stacks or internal RTOS control blocks. **The real accident scene has already been destroyed.** Therefore, the HardFault you see often means innocent code is taking the blame for someone else.

------

## 5. FreeRTOS's Three Safety Nets

FreeRTOS does not assume that you will "never write wrong code." It prepares protection mechanisms in advance.

**1. Stack overflow checking (must be enabled)**

```c
#define configCHECK_FOR_STACK_OVERFLOW  2
```

Once stack overflow is detected, the system immediately enters `vApplicationStackOverflowHook`, and the error occurs at the place closest to the accident. If you do not enable this, you are voluntarily giving up the most important debugging clue.

**2. Stack high-water monitoring (essential in engineering)**

```c
uxTaskGetStackHighWaterMark(NULL);
```

This returns the historical minimum remaining stack space of the task. The correct usage is: run the most complex and extreme business scenarios, observe whether the water mark approaches the danger boundary, and then decide whether to adjust the stack size. This is a stack estimation method truly used in industry.

**3. Malloc failed hook**

```c
#define configUSE_MALLOC_FAILED_HOOK  1
```

When RTOS Heap is exhausted, the system will not quietly return `NULL`. It will immediately enter the hook so the problem is exposed at the earliest point.

------

## 6. MSP: The Stack You Almost Never Pay Attention To

Besides task stacks, there is an even more hidden memory area: **MSP (Main Stack Pointer), the interrupt stack**.

Its characteristics are: shared by all interrupts, not managed by FreeRTOS, and consumed more heavily as interrupt nesting becomes deeper.

Typical traps include: calling `printf` inside interrupts, doing heavy logic inside DMA callbacks, or setting MSP too small while interrupt nesting is deep. These problems often appear as occasional HardFaults that reproduce only under high load or in specific scenarios.

------

## 7. Why Does the System "Run Fine at the Beginning"?

This confuses many people. The reason is: stack overflow happens progressively, heap exhaustion is exposed with delay, and overwriting occurs in places you least suspect.

RTOS system crashes rarely happen instantly. They are more like problems accumulating little by little somewhere you cannot see.

------

## 8. Five Iron Rules for Engineering-Level Memory Use

1. **RTOS Heap and task stacks are two separate systems** - do not mix them up
2. **Give task stacks more rather than gambling with too little** - debugging cost is far higher than a few hundred extra bytes
3. **`printf` is a major stack consumer** - especially `snprintf` with formatting
4. **Stack overflow checking and hooks must be enabled** - they are your most important debugging clues
5. **Before release, trim stacks using high-water data, not by feeling** - data-driven, not guess-driven

These five rules are enough to avoid most "silent crash" problems.

------

## 9. Back to the Overall Philosophy of the RTOS

Returning to the first article's point: FreeRTOS is a system of order, and memory is the boundary of that order.

When you cross that boundary, the RTOS will not punish you immediately, but it will eventually collect the cost at some future moment.

------

## Final Words

Many engineering accidents do not happen because the code is badly written. They happen because system complexity has already exceeded your intuitive control over memory.

An RTOS is not there to make things simple. It is there so that complex systems can still be understood, maintained, and evolved by humans.
