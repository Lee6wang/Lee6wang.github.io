# How a Microcontroller Gets from Power-On to main()

## Preface

After some busy experimenting, we finally wrote our first piece of code, successfully turned on an LED, and completed the hardware world's "Hello World". Everything seems so simple: we write a `void main()`, then start adding our own initialization code and business functions after it.

It is easy to think that the program starts from the `main` function. But is that really true? In practice, it is not.

## The Instant of Power-On (Hardware Layer)

At the moment of power-on, VDD rises to the operating voltage of the STM32. The internal **POR/PDR** circuit, power-on/power-down reset, and **BOR**, brown-out reset if enabled, keep the chip in the reset state until the power supply becomes stable. The reset process follows a similar path.

After power-on or reset, the clock is not yet the external crystal oscillator we usually configure. At this point, the core runs from the default **HSI 8 MHz** clock. The PLL is still disabled, and Flash is also in its default state.

## Boot Medium Selection (Boot Address Space)

After power-on, the board decides where to fetch the **initial stack pointer MSP** and the **reset vector PC**. These two values are determined by the hardware boot configuration. Their initial addresses are `[0x00000000]` and `[0x00000004]`. What exactly is stored at those addresses depends on the selected boot mode, which will be explained below.

![BOOT pins](../assets/figure/2025-11-01/1.png)

Take the STM32F103C8T6 as an example. You can clearly see two pins labeled BOOT0 and BOOT1. On a typical development board, these pins are connected to jumpers by default. The microcontroller reads the levels of these two pins to decide what values should be assigned to MSP and PC.

According to the pin states, the boot options are:

![Boot modes](../assets/figure/2025-11-01/2.png)

The corresponding functions are as follows.

### 1. Flash Boot (Most Common)

After power-on, the MCU maps **main Flash (0x08000000)** to `0x00000000`.

Then `[0x08000000]` is used as MSP, and `[0x08000004]` is used as PC, so execution jumps to `Reset_Handler`.

The program enters `SystemInit()` from the startup file and then reaches `main()`.

### 2. System Memory Bootloader

When **BOOT0=1, BOOT1=0**, the MCU maps internal **System Memory**, a fixed 16 to 32 KB ROM region, to `0x00000000`.

This region contains ST's official **Bootloader**, used for **in-system programming (ISP)**.

Supported communication interfaces depend on the specific device model and may include:

- **USART1 (PA9/PA10)**
- **USART2 (PA2/PA3)**
- **USART3 (PB10/PB11)**
- **USB DFU**, on some models
- **CAN**, on some models

### 3. SRAM Boot Mode (For Debugging)

When **BOOT0=1, BOOT1=1**, the MCU maps **SRAM (0x20000000)** to `0x00000000`.

MSP and PC are then taken from `[0x20000000]`.

## Vector Table and Reset Vector (The First Fetch)

Now, through the selected boot mode, the MCU has obtained the values it needs and assigned them to MSP and PC. From here, continue with the Flash boot mode as the example.

At this time, MSP is taken from `[0x08000000]`, which is in the STM32 internal Flash, and PC is taken from `[0x08000004]`. If we inspect the `startup_stm32f10x_md.s` file in the project, part of it looks like this:

```c
; Vector Table Mapped to Address 0 at Reset
                AREA    RESET, DATA, READONLY
                EXPORT  __Vectors
                EXPORT  __Vectors_End
                EXPORT  __Vectors_Size

__Vectors       DCD     __initial_sp               ; Top of Stack
                DCD     Reset_Handler              ; Reset Handler
                DCD     NMI_Handler                ; NMI Handler
                DCD     HardFault_Handler          ; Hard Fault Handler
                DCD     MemManage_Handler          ; MPU Fault Handler
```

Start with `__initial_sp`. This is the MSP address mentioned earlier. The second entry is where the PC points, which corresponds to a function named `Reset_Handler`.

```c
; Reset handler
Reset_Handler    PROC
                 EXPORT  Reset_Handler             [WEAK]
     IMPORT  __main
     IMPORT  SystemInit
                 LDR     R0, =SystemInit
                 BLX     R0
                 LDR     R0, =__main
                 BX      R0
                 ENDP
```

Here, the program first executes the `SystemInit` function. Look at the function:

```c
void SystemInit (void)
{
#if defined(STM32F100xE) || defined(STM32F101xE) || defined(STM32F101xG) || defined(STM32F103xE) || defined(STM32F103xG)
  #ifdef DATA_IN_ExtSRAM
    SystemInit_ExtMemCtl(); 
  #endif /* DATA_IN_ExtSRAM */
#endif 

  /* Configure the Vector Table location -------------------------------------*/
#if defined(USER_VECT_TAB_ADDRESS)
  SCB->VTOR = VECT_TAB_BASE_ADDRESS | VECT_TAB_OFFSET; /* Vector Table Relocation in Internal SRAM. */
#endif /* USER_VECT_TAB_ADDRESS */
}
```

First look at the upper block. It is related to whether the device has FSMC and larger memory support. For the C8T6, which is a medium-density device, this part can be skipped and does not need much attention.

The lower block is more important. If `USER_VECT_TAB_ADDRESS` is defined, the interrupt vector table uses a user-defined location. Its address is determined by `VECT_TAB_OFFSET`, which is effectively a base address plus an offset from the original vector table. For beginners, it is not recommended to modify this part casually, because it may put the MCU into an abnormal working state. The reason this mechanism exists is to support more advanced use cases, such as bootloaders.

At this point, a question may arise: why is this function called `SystemInit`, but it does not seem to complete clock initialization? The reason is that the microcontroller is still in the early startup state. At this stage, the goal is simply to make sure the chip can run. More accurate clock configuration is needed later when user business logic starts, so the clock initialization is usually performed afterward.

After this function finishes, execution goes to `__main`. Is this already the final `main` function we want? Not yet. This is the hidden entry provided by the ARM C runtime. It initializes the stack and heap, copies `.data`, clears `.bss`, and sets up the C runtime environment. We usually do not see this code directly.

After this step finishes, the system can finally start executing the `main` function.

## Executing the main Function

Below is a typical `main` function generated by CubeMX:

```c
int main(void)
{

  /* USER CODE BEGIN 1 */

  /* USER CODE END 1 */

  /* MCU Configuration--------------------------------------------------------*/

  /* Reset of all peripherals, Initializes the Flash interface and the Systick. */
  HAL_Init();

  /* USER CODE BEGIN Init */

  /* USER CODE END Init */

  /* Configure the system clock */
  SystemClock_Config();

  /* USER CODE BEGIN SysInit */

  /* USER CODE END SysInit */

  /* Initialize all configured peripherals */
  MX_GPIO_Init();
  /* USER CODE BEGIN 2 */

  /* USER CODE END 2 */

  /* Infinite loop */
  /* USER CODE BEGIN WHILE */
  while (1)
  {
    /* USER CODE END WHILE */

    /* USER CODE BEGIN 3 */
  }
  /* USER CODE END 3 */
}
```

Only at this stage does the microcontroller start initializing the clock and then executing the business functions we write later.
