# How I Built a Long-Term Development Environment on a New MacBook

## Foreword

Every time I get a new MacBook, the moment I open the lid is exciting: the system is clean, the desktop is empty, and there are no historical leftovers. But for a developer, this "cleanliness" quickly turns into another task: rebuilding an entire working environment.

Many people understand this as "installing a few apps". Download some apps from the App Store, drag a few `.dmg` files from official websites, and it feels done. But if this machine will work with you for one or two years, or even longer, you will soon find that the real trouble is never just installing software. It is rebuilding your workflow: how to arrange toolchains, where to place projects, how to configure the terminal, and which things deserve automation.

This article records how I set up a development environment from zero on a new MacBook. My usage scenarios are roughly:

- **Code development:** C/C++, Python, and occasional frontend scripts
- **Embedded auxiliary development:** compiling, flashing, serial debugging, and connecting to development boards such as STM32
- **SSH remote development:** connecting to remote Linux servers for building and deployment
- **Scripts and toolchains:** writing automation scripts, data processing scripts, and build flows
- **Documents and knowledge management:** writing technical blogs, organizing project docs, and managing a personal knowledge base

What I want is not a computer with every possible app installed, but a working platform that can support long-term learning, development, and output.

---

## First Clarify the Goal: What Role Should This Machine Play?

Before installing anything, I recommend stopping to think about one question: **what role should this MacBook play in your workflow?**

This question sounds a little abstract, but it directly determines most later choices. For example:

- If you mainly do full-stack web development, the Node.js ecosystem and container tools are core.
- If you do embedded development, serial tools, cross-compilation toolchains, and remote debugging should have higher priority.
- If you also produce content, document tools and directory structure should not be random.

My positioning for this machine is: **the main machine in a modern development workflow**. It does not need to run everything. Some vendor-specific IDEs, special Windows drivers, and old simulation tools do not have to be solved on the Mac. But it should cover at least 80% of my daily development and writing needs, and it should feel smooth.

In other words, I am not pursuing "universal". I am pursuing "efficient and usable for a long time".

## System-Level Base Environment: Build the Foundation First

### Update the System

After getting the new machine, the first thing is to update macOS to the latest version. This is not obsessive behavior. Many later tools have minimum system-version requirements, and Homebrew and Xcode CLI Tools also depend on relatively new system components.

Open "System Settings -> General -> Software Update" and let it finish.

### Install Xcode Command Line Tools

macOS does not include the full development toolchain by default. If you type `gcc` or `git` in the terminal, it will usually prompt you to install this toolset. It provides basic tools such as GCC/Clang, Git, and Make, and many later development tools depend on it.

```bash
xcode-select --install
```

After the dialog appears, click install. It usually takes only a few minutes. After installation, verify it:

```bash
gcc --version
git --version
```

This is not the full Xcode IDE, which is large. It is only the command-line tool collection. If you do not do iOS development, the full Xcode can be skipped for now.

### Install Homebrew

Homebrew is basically the default package manager on macOS. On Linux you may use `apt` or `yum`; on Mac, the corresponding tool is `brew`. Most command-line tools, development libraries, and even some GUI applications can be managed by it.

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

After installation, follow the terminal prompt to add Homebrew to `PATH`. On Apple Silicon Macs, Homebrew is installed in `/opt/homebrew` by default, so the shell configuration usually needs:

```bash
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc
```

Verify it:

```bash
brew --version
```

I put Xcode CLI Tools and Homebrew at the beginning because they are not specific development tools. They are the **infrastructure for almost every later tool**. If these two are not handled well, the following steps will often get stuck.

## General Development Tools: Fill the Toolbox First

After the base environment is ready, install the general tools used by almost all developers. My principle is simple: **install high-frequency essentials first; do not pile up too many things that may be useful someday.**

### Git

Although Xcode CLI Tools includes Git, its version is usually old. Install a newer version with Homebrew:

```bash
brew install git
```

Then configure the basics:

```bash
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"
git config --global init.defaultBranch main
```

If you use GitHub, configure an SSH key so you do not need to enter your password every time you `push`:

```bash
ssh-keygen -t ed25519 -C "your-email@example.com"
cat ~/.ssh/id_ed25519.pub
# Copy the public key and add it to GitHub -> Settings -> SSH Keys
```

### CMake and Ninja

If you write C/C++, CMake is almost unavoidable. Ninja is a lightweight and fast build executor. Together with CMake, it gives a much better build experience:

```bash
brew install cmake ninja
```

### Python

The Python bundled with macOS is usually old and not ideal for development. Installing an independent Python through Homebrew is cleaner:

```bash
brew install python
```

After installation, `python3` and `pip3` can be used directly. If later projects need multiple Python versions, add `pyenv` then. There is no need to make the initial setup too complicated.

One habit I strongly recommend is using virtual environments. One project, one `venv`, avoids most global package conflicts:

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### Node.js

Even if you are not a frontend developer, Node.js is hard to completely avoid now. Many CLI tools, document generators, lint tools, and even some AI toolchains depend on it. I recommend managing versions with `nvm`:

```bash
brew install nvm
```

Then add the `nvm` initialization configuration to `~/.zshrc` according to the installation prompt. After that:

```bash
nvm install --lts
nvm use --lts
node --version
```

### VS Code

Visual Studio Code is now one of the most universal cross-platform code editors. Its ecosystem is mature, and the extensions are rich enough to cover C/C++, Python, remote development, and more.

```bash
brew install --cask visual-studio-code
```

After installation, I usually install these core extensions first:

- **C/C++** from Microsoft: syntax highlighting, IntelliSense, and debug support
- **Python** from Microsoft: full Python development support
- **Remote - SSH:** edit remote code from local VS Code
- **GitLens:** better Git history and blame view
- **Markdown All in One:** convenient for documents and blogs

VS Code can also be launched directly from the command line. Typing `code .` in the terminal opens the current directory. If this command is not available, press `Cmd+Shift+P` in VS Code and search for "Shell Command: Install 'code' command in PATH".

### iTerm2

The built-in macOS Terminal works, but its features are basic. iTerm2 has become the default choice for many Mac users. Split panes, tabs, custom shortcuts, and color schemes are more mature and feel better over long-term use.

```bash
brew install --cask iterm2
```

Configure a comfortable shell environment as well. I usually use Oh My Zsh to manage zsh configuration:

```bash
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"
```

### Other Common Command-Line Tools

I recommend installing the following tools at once because they are used very frequently:

```bash
brew install tmux wget curl htop tree jq ripgrep fd bat
```

Why keep these tools:

- **tmux:** terminal multiplexer, especially useful when SSHing into servers; sessions survive disconnection
- **wget / curl:** downloading files and debugging HTTP requests
- **htop:** process monitor that is much nicer than the built-in `top`
- **tree:** view directories as a tree, useful for understanding project structure
- **jq:** command-line JSON processor, essential when debugging API responses
- **ripgrep (`rg`):** much faster full-text search than grep
- **fd:** a more convenient file-search tool than find
- **bat:** a `cat` replacement with syntax highlighting

Each tool is small by itself, but together they make the command-line environment much smoother.

## Additional Tools Related to My Development Direction

The tools above are general configuration. The following part depends more on personal direction. My work leans toward embedded and hardware-related development, so the additional tools mainly serve that scenario.

### Serial Debugging

For embedded development, serial ports are almost the most basic debugging method. On Mac, common serial tools include `minicom` and `screen`:

```bash
brew install minicom
```

After plugging in a USB-to-serial cable, check whether the device is recognized:

```bash
ls /dev/tty.usb*
```

Then connect:

```bash
minicom -D /dev/tty.usbserial-xxxx -b 115200
```

If you prefer a GUI, CoolTerm is also a good option and can be downloaded directly from the official website.

### OpenOCD

OpenOCD is an open-source on-chip debugging tool. It supports JTAG/SWD and can work with debuggers such as ST-Link and J-Link to flash and debug chips like STM32:

```bash
brew install open-ocd
```

One caveat: OpenOCD on Mac may occasionally encounter driver compatibility issues, especially when some debuggers require extra USB drivers. When this happens, the answer usually needs to be found in the official documentation of the specific debugger.

### ARM Cross-Compilation Toolchain

If you plan to compile firmware for ARM Cortex-M on Mac, the ARM GCC toolchain is basically required:

```bash
brew install --cask gcc-arm-embedded
```

Verify after installation:

```bash
arm-none-eabi-gcc --version
```

With CMake and Ninja, most embedded projects can be compiled on the Mac.

### Advantages and Boundaries of Mac in Embedded Development

To be honest, Mac is not the traditional main platform for embedded development. Many vendor IDEs, such as Keil MDK and IAR, support only Windows. Some proprietary flashing tools and drivers are also basically Windows-only.

But Mac is not incapable. Its advantages are also clear:

- **Good compatibility with modern development workflows:** open-source toolchains such as GCC + CMake + OpenOCD run smoothly on Mac, especially for projects using Makefile/CMake.
- **Terminal experience is much better than Windows:** zsh + iTerm2 + tmux gives high daily efficiency.
- **Strong SSH remote development:** connecting to Linux build servers is very convenient.
- **Smooth switching with other development work:** write firmware, switch to Python for a test script, then switch to Markdown for documentation without changing machines.

My strategy is clear: **Mac handles daily coding, compiling, debugging, and documents. If a scenario must depend on Windows-specific tools, use a virtual machine or another Windows machine.** There is no need to force every task into one machine.

## Workspace and Directory Structure: Do Not Ignore File-System Design

When setting up an environment, many people only care about what software is installed. But another equally important thing is often ignored: **where files should be placed and how they should be organized.**

At the beginning, this may not seem important. After half a year, projects scattered across the desktop, Downloads, and temporary folders become painful to search. If the directory structure is designed clearly from the start, it saves a lot of mental overhead later.

The directory structure I often use is roughly:

```text
~/
|-- Workspace/          # All code projects
|   |-- embedded/       # Embedded-related projects
|   |-- python/         # Python projects
|   |-- scripts/        # Utility scripts
|   |-- web/            # Frontend / Web projects
|   `-- playground/     # Temporary experiments and learning
|
|-- Knowledge/          # Knowledge base and study notes
|   |-- notes/          # Technical notes
|   |-- references/     # References and manuals
|   `-- reading/        # Reading notes
|
|-- Creative/           # Content output
|   |-- blog/           # Blog articles
|   |-- docs/           # External documents
|   `-- slides/         # Presentations
|
|-- Archive/            # Archived completed or paused projects
|
`-- Inbox/              # Temporary collection area for files to sort later
```

Several design ideas matter to me:

- **Workspace is divided by technical direction**, not randomly by project name. This lets you quickly locate project categories.
- **Knowledge and Creative are separated.** Knowledge is input, such as learning and accumulation; Creative is output, such as writing and publishing.
- **Inbox is a buffer.** Temporary downloads, files sent by others, and things whose location is not decided yet can go into Inbox and be sorted regularly.
- **Archive is cold storage.** Completed or inactive projects move here. They are not deleted, but they no longer occupy attention in the main directories.

This structure can be created with one command:

```bash
mkdir -p ~/Workspace/{embedded,python,scripts,web,playground} \
         ~/Knowledge/{notes,references,reading} \
         ~/Creative/{blog,docs,slides} \
         ~/Archive \
         ~/Inbox
```

This is not a sophisticated technique, but it is a key step in turning a machine from "a pile of files" into "an organized working platform".

## What Can This Machine Do After Setup?

At this point, the environment is basically ready. Looking back, this MacBook can now handle:

**Writing C/C++ code:** GCC/Clang compilers, CMake + Ninja build system, and VS Code C/C++ extensions provide completion and debug support. Local projects and embedded cross-compilation are covered.

**Writing Python:** independent Python environment, `venv`, and `pip` are available. Data processing scripts, automation tools, and simple web services are all fine.

**Writing tool scripts:** a complete shell environment, command-line tools such as `jq`, `ripgrep`, and `fd`, plus Python, are enough for daily automation, data processing, and batch operations.

**Managing projects:** Git handles version control, a clear directory structure handles project categories, and VS Code handles browsing and editing. Multiple parallel projects are less likely to become chaotic.

**Connecting development boards:** serial tool `minicom`, debug tool OpenOCD, and ARM cross-compiler toolchain are installed. Development boards can be compiled, flashed, and debugged.

**Remote SSH development:** iTerm2 + tmux manages terminals, and VS Code Remote SSH handles remote editing. Working on a remote Linux server feels close to local development.

**Writing documents and blogs:** structured knowledge-base directories, Markdown editing support, and note templates provide places for technical notes and formal blog articles.

This looks like a list of tools, but more accurately, it is a workflow whose parts cooperate with each other.

## Real Problems and Boundaries: Mac Is Not a Universal Answer

Objectively, Mac has many advantages in development, but its boundaries are also clear. Ignoring these boundaries usually creates extra trouble.

**Vendor-specific tools depend on Windows.** In embedded development, Keil MDK and IAR Embedded Workbench are typical Windows-only tools. Some chip-vendor configuration tools, such as clock configurators and pin assignment tools, are also Windows-only. If a project strongly depends on them, Windows cannot be avoided.

**Some hardware drivers are incomplete.** Most USB-to-serial chips, such as CH340, CP2102, and FTDI series, have Mac drivers, but occasionally a debugger or programmer cannot be recognized on Mac.

**Virtual machines and dual systems have extra cost.** Running Windows virtual machines on Apple Silicon Mac through Parallels or UTM is feasible, but performance and compatibility are not as good as native Windows. ARM Windows still has gaps for some x86 applications.

**My response strategy has three layers:**

1. **Mac as the main machine:** daily coding, compiling, debugging, documents, and blogs are all done on Mac.
2. **Virtual machine as supplement:** occasional Windows tools are handled in a VM.
3. **Independent Windows machine as fallback:** heavy Windows-dependent scenarios, such as some mass-production flashing tools, are handled by a dedicated Windows machine.

Do not be obsessed with making one machine handle everything. Recognize what each tool is good at and distribute tasks clearly. This is usually the most efficient approach.

## Conclusion

Looking back at this article, it covers a lot, from system update and Homebrew installation to directory structure and terminal environment. But if I summarize the core in one sentence:

**You are not building a computer full of software; you are building a working system that can support long-term learning, development, writing, and output.**

Software will update and tools will be replaced, but the workflow, directory structure, and habits you slowly build will stay with you. They are the real "environment".

Everyone's development direction and habits differ. What I share here is only a setup that works smoothly for me. You can adjust it based on your own needs, remove unnecessary parts, and add your own tools and configuration. The important thing is to **design this environment consciously from the beginning, instead of installing whatever you need at the moment and eventually turning the machine into chaos.**

I hope this article helps if you are configuring a new MacBook.
