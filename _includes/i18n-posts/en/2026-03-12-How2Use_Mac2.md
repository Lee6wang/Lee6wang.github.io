# Stop Turning Your Desktop into a Dump: Workspace Partitioning and Scripted Setup for Development

> Your file system is not only a place to store code. It is the physical projection of your whole way of working. If every file search depends on luck, and every new project makes you hesitate about where to put it and what to name it, the problem is not memory. The problem is the system.

---

## Foreword: Why Does Your Computer Get Messier the Longer You Use It?

Almost every developer goes through this stage: when the new computer arrives, everything is clean. Half a year later, there are dozens of files on the desktop, thousands of records in Downloads, projects scattered under three or four parent directories, and several folders named `test`, `demo`, or `untitled` sitting near the root. You know a file is "probably somewhere", but finding it means opening multiple directories and searching globally several times.

This is usually not a habit problem. It is a structure problem from the very beginning.

Many people's default way of using a computer is similar: desktop as entrance, Downloads as warehouse, Documents as storage room. For short-term tasks, this is acceptable. But once the work cycle becomes longer, the problem becomes obvious. Desktop and Downloads have no semantics. They are only default collection boxes provided by the system. They cannot express "what kind of work this belongs to" or "what stage this item is in".

If you work on code development, document writing, knowledge management, study notes, and content output at the same time, the file system is no longer just a place to install software. It is **infrastructure for organizing information and projects**. If this infrastructure is not designed well, project management, knowledge accumulation, and efficiency improvement become much harder.

The core question this article wants to solve is: how to build a **stable, maintainable, and extensible workspace structure** for a development machine, and then fix it in place with scripts.

## Overall Idea of Workspace Partitioning

### Core Principle: Partition by Responsibility, Not File Type

When many people organize their computers, their first reaction is to split by file type: one folder for documents, one for images, one for code. This sounds reasonable, but it often fails quickly in practice. A complete project usually contains code, documents, images, and configuration files at the same time. If they are split by type, the context of the same task is scattered.

A better method is **partitioning by responsibility**. Each top-level directory should represent a kind of work state or work nature, not a file format.

Recommended top-level structure:

```text
~/
|-- Workspace/       # Active projects
|-- Knowledge/       # Long-term knowledge base and notes
|-- Creative/        # Creation and external output
|-- Archive/         # Historical archives
`-- Inbox/           # Temporary receiving area
```

### Responsibilities of Each Area

**Workspace** is the main battlefield. All active development projects and work tasks go here. Its most important feature is "active": content in it is repeatedly opened and modified recently.

**Knowledge** is the knowledge warehouse. Technical notes, reading excerpts, learning reflections, API references, and domain knowledge that do not necessarily belong to a specific project but may be repeatedly searched later should be stored here.

**Creative** is the creation space. Blog articles, technical talks, open-source documentation, portfolio materials, and other output-oriented content are drafted, edited, and polished here.

**Archive** is historical storage. Finished projects, paused materials, and things that are no longer active but may be needed later can be archived here. Its value is keeping other directories clean.

**Inbox** is a temporary buffer. Files sent by others, newly downloaded materials, and things whose destination is not yet clear go here first. Later, clean it regularly: archive what should be archived and delete what should be deleted.

### Why the Top Level Should Not Exceed Five or Six Directories

The top-level directory is the first navigation layer you see every day in the terminal or file manager. If there are more than a dozen directories here, you still need to scan before deciding where to go. That is not much better than having no design.

Keeping the top level around five directories is usually enough for most personal work scenarios, and every name should immediately suggest what belongs there.

The key is not extreme detail. It is **clear boundaries**. If you frequently hesitate between two directories, their responsibilities probably overlap and should be redrawn.

## Second-Level Structure: Make the Partitions Usable

The top-level structure is the framework. The second level determines daily comfort.

### Workspace: Split by Project

```text
Workspace/
|-- robotics-hand/          # Dexterous hand project
|-- motor-driver-stm32/     # STM32 motor driver
|-- personal-site/          # Personal website
|-- freelance-client-a/     # Client project
`-- experiment-can-bus/     # Experimental project
```

Each subdirectory is an independent project, and the project name is the directory name. There is no need to add another layer by language, framework, or work type, because the project itself is the natural organization unit. Inactive projects should move to Archive instead of staying in Workspace.

### Knowledge: Split by Topic Domain

```text
Knowledge/
|-- embedded/               # Embedded development
|-- robotics/               # Robotics
|-- programming/            # General programming knowledge
|-- hardware/               # Hardware design
|-- tools/                  # Tools and environment
|-- reading-notes/          # Reading notes
|-- papers/                 # Paper reading
|-- courses/                # Course notes
|-- daily/                  # Daily reports
|-- weekly/                 # Weekly reports
`-- monthly/                # Monthly summaries
```

This stores **general knowledge separated from specific projects**. For example, if you learn some I2C communication experience while working on a dexterous hand project, implementation details can stay in the project directory, but general I2C notes belong in `Knowledge/embedded/` because they may be reused in future projects.

Project materials and long-term knowledge should be separated. Projects end, but knowledge accumulation usually does not. If all notes are scattered inside project directories, they disappear from daily view once the project is archived.

### Creative: Split by Stage

```text
Creative/
|-- drafts/                 # Drafts
|-- published/              # Published output
|-- assets/                 # Asset library, images, templates, and so on
`-- talks/                  # Talks and sharing materials
```

Creative work naturally has a life cycle: collecting assets, drafting, editing, and publishing. Mixing drafts with finished output is common and error-prone. Drafts are intermediate and change frequently. Published work should be stored stably.

### Archive: Split by Year

```text
Archive/
|-- 2024/
|-- 2025/
`-- 2026/
```

Archive does not need a complex classification system. Splitting by year is usually enough because memory of old work is often time-based: "that was probably something I did last year." If one year has too much content, add subdirectories under that year later.

### Inbox: Keep It Simple

```text
Inbox/
|-- downloads/              # Download buffer
`-- incoming/               # Files sent by others
```

Inbox has one rule: **clear it regularly**. It is a buffer, not a warehouse. If a file stays in Inbox for more than two weeks, it usually has only two outcomes: archive it or delete it.

## File Naming Rules: More Important Than Classification

Directory structure answers "where to put it". Naming rules answer "how to find it". In real work, searching usually happens more often than placing.

### Why Naming Rules Are Extremely Important

macOS Spotlight, Linux search tools, and VS Code file search all depend heavily on filename quality. If a file is named "new document.docx" or "final_v2_real_final(1).pdf", no search tool can really save you.

### Recommended Naming Pattern

For time-related content such as notes, daily reports, blogs, and archived files:

```text
YYYY-MM-DD-topic-description.ext
```

Examples:

```text
2026-03-12-i2c-multiplexer-debug-note.md
2026-03-10-seed-round-pitch-deck-v2.pptx
2026-02-robotic-hand-patent-materials.zip
```

For project directories, use concise English kebab-case names:

```text
robotics-hand-firmware
stm32-can-driver
personal-blog-site
```

### Naming Focus for Different Content

**Documents and notes:** date first, topic later. Sorting by filename naturally gives chronological order.

**Project directories:** descriptive kebab-case without date. Time information is managed by Git and the file system.

**Images and assets:** include source and purpose, such as `robotics-hand-pcb-layout-v3.png`, not `screenshot_2026.png`.

**Archive files:** date + project name + short description, such as `2025-12-motor-driver-project-final.tar.gz`.

### High-Risk Naming Methods

Avoid these completely:

- Meaningless names such as `new-document`, `untitled`, `test`, `demo`, and `tmp`
- Version chaos such as `final`, `final2`, and `real-final`
- Long filenames with spaces that are difficult to quote and escape in the command line
- Pure numeric names such as `001.md` and `002.md`, which become unreadable outside their folder context

## Why Scripted Setup Is Needed

Now you have a clear directory structure and naming convention. The next question is: how do you make it run consistently?

The biggest problem with manually creating directories is **inconsistency**. Today you remember to create `docs/` and `references/` in a project. Next month, when rushing, you may forget. Manual operations also have a hidden cost: every time you need to rethink how the structure should be created.

The value of scripting is simple: **consistent, fast, reusable**. One shell function can generate a standard structure in seconds. It is faster and more stable than doing it manually, and the output is the same every time.

Shell and zsh functions are well suited for this. No extra software and no new language are needed. A few command lines can handle the repetitive work.

## Basic Directory Auto-Creation Script

The following script initializes the whole workspace structure in one command. Run it once and the five main areas and their subdirectories are created.

```bash
#!/bin/bash
# setup-workspace.sh
# Initialize workspace partition structure in one command.
# Usage: bash setup-workspace.sh [root-dir]
# Default root directory is ~/

ROOT="${1:-$HOME}"

echo "Initializing workspace structure: $ROOT"

# ===== Workspace: active projects =====
mkdir -p "$ROOT/Workspace"

# ===== Knowledge: long-term knowledge base =====
mkdir -p "$ROOT/Knowledge"/{embedded,robotics,programming,hardware,tools}
mkdir -p "$ROOT/Knowledge"/{reading-notes,papers,courses,daily,weekly,monthly}

# ===== Creative: output area =====
mkdir -p "$ROOT/Creative"/{drafts,published,assets,talks}

# ===== Archive: historical archive =====
YEAR=$(date +%Y)
mkdir -p "$ROOT/Archive/$YEAR"

# ===== Inbox: temporary receiving area =====
mkdir -p "$ROOT/Inbox"/{downloads,incoming}

# Generate README for every top-level directory
for dir in Workspace Knowledge Creative Archive Inbox; do
  README="$ROOT/$dir/README.md"
  if [ ! -f "$README" ]; then
    echo "# $dir" > "$README"
    echo "" >> "$README"
    echo "Refer to the workspace partition design document for this directory's purpose." >> "$README"
  fi
done

echo "Workspace initialization completed."
echo ""
echo "Directory preview:"
find "$ROOT"/{Workspace,Knowledge,Creative,Archive,Inbox} -maxdepth 2 -type d 2>/dev/null | \
  sed "s|$ROOT/||" | sort | while read -r line; do
    depth=$(echo "$line" | tr -cd '/' | wc -c)
    indent=$(printf '%*s' $((depth * 2)) '')
    basename=$(basename "$line")
    echo "${indent}|-- $basename/"
  done
```

Several details are worth noting:

- **`mkdir -p`:** recursively creates directories. If the parent directory does not exist, it creates it. If the directory already exists, it does not report an error. This guarantees idempotence.
- **`README.md`:** placing a README in each top-level directory leaves a purpose note for your future self.
- **`.gitkeep`:** if you need Git to track empty directories, create a `.gitkeep` file inside them. Git does not track empty directories by itself.

## Custom Command System: Seven Complete Categories

Now comes the core of the article. The directory structure is the skeleton. The command system below is what makes it move.

I split custom commands into seven categories. Each category solves a specific problem, and together they form a complete terminal workflow.

### Command File Organization

Before writing commands, decide how to organize the command files. Do not put everything into `~/.zshrc`. Split commands by module:

```text
~/.zsh/custom/
|-- 01-navigation.zsh      # Directory navigation
|-- 02-workspace-view.zsh  # Workspace inspection
|-- 03-project.zsh         # Project creation
|-- 04-knowledge.zsh       # Knowledge and writing templates
|-- 05-finder.zsh          # Quick open directories
|-- 06-maintenance.zsh     # Configuration and maintenance
`-- 07-utils.zsh           # Automation helpers
```

Add automatic loading logic to `~/.zshrc`:

```bash
# Append to ~/.zshrc
# Load all custom command modules automatically
if [ -d ~/.zsh/custom ]; then
  for config_file in ~/.zsh/custom/*.zsh(N); do
    source "$config_file"
  done
fi
```

**Goal:** inspect files and directory structure quickly.

```bash
# ----- Tree view -----
# showtree: show a directory tree
# Usage: showtree [directory] [depth]
# Example: showtree ~/Knowledge 2
showtree() {
  local dir="${1:-.}"
  local depth="${2:-2}"

  if command -v tree >/dev/null 2>&1; then
    tree -L "$depth" "$dir"
  else
    find "$dir" -maxdepth "$depth" -type d | sort
  fi
}
```

### Category 3: Project Creation Commands

**Goal:** create a standard project structure.

```bash
# ~/.zsh/custom/03-project.zsh
# ============================================
#  Project creation commands
# ============================================

# mkproj: create a standard project structure
# Usage: mkproj <project-name>
# Example: mkproj robotics-hand-v2
mkproj() {
  if [ -z "$1" ]; then
    echo "Usage: mkproj <project-name>"
    return 1
  fi

  local project_name="$1"
  local project_dir="$HOME/Workspace/$project_name"

  if [ -d "$project_dir" ]; then
    echo "Project already exists: $project_dir"
    return 1
  fi

  mkdir -p "$project_dir"/{src,include,docs,tests,scripts,references}

  cat > "$project_dir/README.md" << EOF
# $project_name

## Project Overview

## Directory Description

- src/: source code
- include/: header files or public interfaces
- docs/: project documents
- tests/: tests
- scripts/: utility scripts
- references/: reference materials

## TODO

- [ ] Initialize project
EOF

  touch "$project_dir/src/.gitkeep"
  touch "$project_dir/include/.gitkeep"
  touch "$project_dir/docs/.gitkeep"
  touch "$project_dir/tests/.gitkeep"
  touch "$project_dir/scripts/.gitkeep"
  touch "$project_dir/references/.gitkeep"

  echo "Project created: $project_dir"
  cd "$project_dir" || return
}
```

### Why These Subdirectories Are Worth Keeping

`src/` and `include/` provide code boundaries. `docs/` keeps project documents close to the code. `tests/` encourages verification from the beginning. `scripts/` stores repeated project actions. `references/` keeps vendor manuals, screenshots, protocol documents, and other background materials with the project.

### The Core Value of Templates

Templates do not solve "whether the layout looks good". They lower the threshold for recording. Many times, people are not unable to write. They are blocked by a blank file and do not know how to start. A template skips that step: the file exists, the structure exists, and you only fill in content.

**Fixed location + fixed naming + fixed structure** is powerful. Notes, daily reports, blog drafts, and paper records become predictable. Half a year later, you know where they are, what they are called, and what you will roughly see after opening them.

**Goal:** open common directories in Finder on macOS or the file manager on Linux, connecting GUI workflow and terminal workflow naturally.

```bash
# ============================================

# Detect open command: macOS uses open, Linux uses xdg-open
if [[ "$(uname)" == "Darwin" ]]; then
```

These commands are simple but high-frequency. After finishing a terminal operation, you may want to drag a file in Finder, view an image, or open a PDF. One `ows` is often much faster than clicking through several layers manually.

```bash
  oblog / opublished           -> open creative directories

Maintenance
  sz                           -> reload ~/.zshrc
  helpme                       -> show this manual

Utilities
  mkgitkeep [dir]              -> add .gitkeep to empty directories
  mkreadme [title]             -> create README.md in current directory
  today                        -> print today's date

=========================================
HELPEOF
}
```

`helpme` is the index of the whole command system. Once commands increase, you will forget them too. This entry makes them searchable. After adding every new command, update `helpme` immediately. Maintaining the index is cheap; not finding a command later is expensive.

### Category 7: Automation Helper Commands

**Goal:** reduce repetitive mechanical actions with small general-purpose helper functions.

```bash
# ~/.zsh/custom/07-utils.zsh
# ============================================
#  Automation helper commands
# ============================================

# ----- mkgitkeep: add .gitkeep to empty directories -----
# Usage: mkgitkeep [directory]
# If no directory is specified, add .gitkeep to all empty subdirectories under current directory.
mkgitkeep() {
  if [ -n "$1" ]; then
    mkdir -p "$1"
    touch "$1/.gitkeep"
    echo "Added: $1/.gitkeep"
  else
    local count=0
    find . -type d -empty ! -path '*/\.*' | while read -r dir; do
      touch "$dir/.gitkeep"
      echo "  + $dir/.gitkeep"
      count=$((count + 1))
    done
    echo "Processed empty directories."
  fi
}

# ----- mkreadme: create README.md in current directory -----
# Usage: mkreadme [title]
# If no title is given, use current directory name.
mkreadme() {
  local title="${1:-$(basename "$(pwd)")}"
  local filepath="./README.md"

  if [ -f "$filepath" ]; then
    echo "README.md already exists."
    ${EDITOR:-code} "$filepath"
    return
  fi

  cat > "$filepath" << EOF
# $title

**Created:** $(date +%Y-%m-%d)

## Overview

## Directory Description

## Notes

EOF

  echo "README.md created."
  ${EDITOR:-code} "$filepath"
}

# ----- today: print today's date string -----
today() {
  echo "$(date +%Y-%m-%d)"
}

# ----- confirm: common confirmation prompt -----
confirm() {
  local prompt="${1:-Confirm?}"
  echo -n "$prompt [y/N] "
  read -r answer
  [[ "$answer" =~ ^[Yy]$ ]]
}
```

Each helper is small, but they appear frequently. `mkgitkeep` solves Git's old problem of not tracking empty directories. `mkreadme` lets you create a standard README in one second. `today` looks almost unnecessary, but when building filenames in the terminal, `$(today)` is smoother than manually typing the date.

## How Workspace Partitioning and Scripted Setup Form a Closed Loop

Now look back at how the whole system connects:

**Directory structure** solves "where to put things". Every type of content has a clear home, so you do not decide from scratch every time.

**Naming rules** solve "how to find things". Stable naming patterns make search reliable and sorting meaningful.

**Automation commands** solve "how to create things quickly". From idea to execution, you do not need to manually assemble the structure every time.

**Templates** solve "how to start". Once the file is opened, you can start writing instead of staring at a blank page.

Together, these four form a complete workflow. Without directory structure, files scatter. Without naming rules, searching depends on luck. Without automation, every start repeats manual setup. Without templates, many records never start.

More importantly, this system is not static. It runs dynamically. Projects begin in Workspace, knowledge settles into Knowledge, outputs take shape in Creative, completed work goes into Archive, and new incoming files enter Inbox first before being sorted.

```text
Inbox (receive) -> Workspace (do) -> Creative (output) -> Archive (store)
                         |
                         v
                    Knowledge (learn)
```

This is not only a static directory diagram. It is a route map for daily work.

## Real Reminders and Boundaries

Before actually using the system, several practical reminders matter:

**Do not design it too complex at the beginning.** Five top-level directories plus one or two levels of subdirectories are enough. Overdesigned directory systems are like overdesigned code: if maintenance cost is too high, you will abandon them.

**Do not automate for the sake of automation.** If you create one project per month, manual `mkdir` may be acceptable. Scripting is justified when you do something often and the steps are similar each time.

**Do not pursue perfect classification.** Some files will always be hard to place. That is normal, and Inbox exists for this reason. The important thing is not perfect placement of every file, but most files being in predictable locations.

**The real key is consistent use.** A plain system you use every day is more valuable than a sophisticated one you abandon after three days. Choose a complexity level you can keep using.

**Add commands gradually.** You do not need all seven categories on day one. Start with the highest-frequency commands, such as navigation and `mkproj`. After using them, you will naturally know what else is needed.

## Conclusion

What you build is not only a group of folders. It is a **personal work infrastructure**.

Code projects move forward in Workspace. Technical knowledge settles in Knowledge. Creative work moves from draft to publication in Creative. Finished work enters Archive and waits quietly until it is needed again. Commands such as `mkproj`, `mknote`, `mkdaily`, `mkweekly`, and `mkmeeting` are essentially tools you build for yourself, turning daily work from "first figure out how to organize this" into "start doing it directly".

A good workspace partition design eventually makes your computer more orderly and easier to use the longer you use it. Spending one afternoon seriously designing it is worthwhile because the return can last for years.
